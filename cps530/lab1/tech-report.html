ITEM 404-56: SECURITY DETAILS

The webpage is protected with a snazzy-looking input form. Using some simple JS, we can check if the given username and password match the correct credentials. It's horribly insecure, all it takes is some madlad to press F12 and the whole world comes crashing down around us. The validation function is given below. From one glance, it should be glaringly obvious why this form of 'security' isn't ideal. 

function validate(){
	if (document.getElementById("username").value=="the404" && document.getElementById("password").value=="insecure") {
		document.getElementById("login-form").style.display = "none";
		document.getElementById("content").style.opacity = 1;
		document.body.style.background.color = "black";
	}
	else{
		document.getElementById("pass-wrong-msg").style.display="block";
		document.getElementById("msg-ok").focus();
		document.getElementById("username").value = "";
		document.getElementById("password").value = "";
		document.getElementById("err-sound").pause();
		document.getElementById("err-sound").currentTime = 0;
		document.getElementById("err-sound").play();
	}
}

ITEM 404-64: <PRE> TAG

Now, I know what you're saying. We're supposed to use that new tag in our actual page, not our technical report.
This actually kind of happened by accident though. You see, I've been doing websites for a few years now and I've used pretty much any tag that comes to mind, and the one's that don't wouldn't really have a place on the page for this lab, no good way to put them to use without going really far out of my way to incorporate them. (You could say I've already done that, though. I mean, just look at it.)
Regardless, there isn't much new to me, but when writing this report I found myself using the <pre> tag to show nicely formatted code blocks, like the JS and HTML printouts. As I'm sure you're aware, the tabs and newlines you type in your editor mean nothing to the browser. Normally you'd need a mess of <br>s and whatnot to get things looking good, but I discovered the <pre> tag is used to define preformatted text, which means I can throw in some nicely formatted code and it'll actually render as typed. Pretty nifty.

ITEM 404-78: DIFFERENCES BETWEEN PLATFORMS

Font sizes vary depending on the platform. Text seems bigger on mobile devices, thanks to responsive font sizes. Some elements are rendered with these by default, while some aren't. This makes some elements seem like they have small text on mobile devices. This is easily rectified, but I just didn't bother to, as I needed something to talk about here. Other than that, nothing more than the usual spacing and font-rendering differences are present, and those are pretty minor.
